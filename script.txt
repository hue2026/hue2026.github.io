[Duration: ~3 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[As students enter, display the title slide with game pad icons. Begin
with high energy, genuine excitement.]

"Good morning / afternoon, game designers! Welcome to a very special
lesson. Today, we are going to answer a question that every single
one of you has probably wondered while playing your favorite games:
How do they actually make these games work?"

"Think about it. When you play Roblox, or Mario, or Minecraft, or anything
else — there is something invisible happening underneath. Something that
makes the characters move, the music play, the collisions happen, and the
rules work smoothly every single time. That invisible thing is called a
game engine."

"By the end of today, you will know what a game engine is, you’ll meet
three real ones that real developers use — Scratch, Unity, and Godot —
and you’ll even start building your own very first game. You are all
going to think like game designers today. Let’s jump in."

💡 [DEEP CONTEXT — deliver this slowly]:
"Here’s something incredible: every video game you have ever loved is not
magic. It is a giant set of instructions running on a computer. But writing
those instructions from scratch every time would take forever. Imagine if
every time someone made a new game, they had to rewrite how the screen
shows images, or how sound comes out of the speakers, or how two objects
know they touched. That would be like re-inventing the wheel for every car!

A game engine is a toolbox that solves all those common problems so game
creators can focus on the fun parts — their characters, their world, their
story. Today we will open that toolbox and look inside."

movement, rules, graphics, sound — that is exactly what a game engine
handles. You already knew more than you thought you did."

SLIDE 4 — KEY VOCABULARY
[Duration: ~5 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Before we go any further, let’s arm ourselves with five powerful words.
Only five. But if you learn these five words, you can talk about game
creation like a pro. Say each one after me."

[Go through each word on the slide. Show the definition, then elaborate.
Point to the card as you speak.]

"WORD 1: GAME ENGINE — software that helps build games. It handles
graphics, physics, sound, and lots more. 💡 Think of it as the brain and
skeleton of your game. It gives you the basic structure so you can focus
on the creative stuff — like designing your characters and levels."

"WORD 2: SCRATCH — a drag-and-drop coding environment made by MIT. It’s
perfect for beginners and for telling stories. 💡 Under the hood, Scratch
is still a real game engine, but simplified. It teaches you the logic of
programming without having to type complicated code. Every block you snap
together is actually a line of code in disguise."

"WORD 3: UNITY — a professional engine used to make 2D and 3D games. Many
famous games like Among Us and Cuphead were made with Unity. 💡 Unity is
like a grown-up version of what we’ll do in Scratch, but much more
powerful. It uses a real programming language called C#. Professionals
love it because it can handle huge worlds and complex physics."

"WORD 4: GODOT — pronounced 'guh-DOH.' It’s totally free and open-source,
meaning anyone can look at its code and change it. It’s lightweight and
great for learning real engine logic. 💡 Godot has its own easy scripting
language called GDScript that looks a lot like Python. Many indie game
developers choose it because it’s free and friendly."

"WORD 5: ITERATIVE DESIGN — a cycle: make a version, test it, get feedback,
and improve. 💡 This is the secret superpower of all good game creators.
No one makes a perfect game on the first try. They build something rough,
see what works, fix what doesn’t, and keep polishing. The best games in
the world went through this loop hundreds or even thousands of times."

"BONUS WORD: SPRITE — any object in a game: your character, an enemy,
a coin, a power-up. Everything you see moving is a sprite."

✋ [Quick check: Point at random students and ask, 'What is a sprite?'
   'What engine did we say is free and open-source?' Keep it fast and fun.]

SLIDE 5 — WHAT IS A GAME ENGINE? (THE CAR ANALOGY)
[Duration: ~4 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[Display the slide with the car image]

"Alright, picture this. You’re building a car. You could go into a field,
dig up metal, shape it into an engine by hand, then build wheels, then
figure out brakes... OR you could use an engine that someone already
designed, and just focus on the color, the seats, the cool sound system."

"A game engine is exactly like a car engine. It’s the hidden machinery
that makes everything run. It already knows how to do the heavy lifting —
like making things move smoothly, checking if two objects have bumped into
each other, playing sounds at the right time, and showing all the graphics
on screen. Without it, a game developer would have to write millions of
lines of code just to get a character to walk across the screen."

💡 [DEEP EXPLANATION]:
"In a professional engine like Unity or Godot, the engine provides a
'scene graph' — a hierarchy of all the objects in the game. It runs a
'game loop' that updates everything dozens of times per second: check
inputs, move objects, detect collisions, render graphics, repeat. The
developer just plugs their own code into that loop. It’s like the
engine knows how to breathe, and you tell it how to dance."

"Without a game engine, today we would spend all class just trying to make
a dot appear on screen. Instead, with Scratch, we’ll make a character
move in about five minutes. That’s the power of using an engine."

💡 [SYNTHESIS]:
"Here’s the key idea: different game engines are like different kinds of
toolboxes. A beginner’s toolbox has big, colorful, easy-to-use tools.
A professional’s toolbox has more tools, but they’re more complicated.
No single engine is 'best' — it depends on what you want to build and
how much time you want to spend learning. Today, we start with Scratch,
the friendliest toolbox in the world."

[Duration: ~5 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Now let’s zoom in on our first engine: Scratch! Raise your hand if
you’ve ever used Scratch before. [Note hands] Great! You’ll be able
to help others. If you haven’t, no problem — it’s built for beginners."

[Point to the Scratch editor breakdown on the slide]

"Scratch has three main areas:
  1. The Block Palette — this is where you find all the code blocks.
     They're color-coded by type: motion, looks, sound, events...
  2. The Sprite Area — here you see all the characters and objects
     in your game. You can add, remove, and edit sprites.
  3. The Stage — where your game actually plays. The big white area
     on the right."

💡 [BEHIND THE CONCEPT]:
"Scratch might look like just snapping colorful blocks, but what you’re
really doing is something called 'event-driven programming.' Each block
is a command, and when you connect them, you're building a sequence of
instructions that the computer will follow. The 'green flag' block is
the event that starts everything. This is the same fundamental logic
used in every programming language on Earth — just wrapped in a friendly
package."

"You can use Scratch right now at scratch.mit.edu. It’s free and runs in
a web browser, no installation needed. In a few minutes, we’ll all build
a simple game together. But first, let’s meet the other two engines."

SLIDE 8 — UNITY
[Duration: ~4 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[Display the Unity slide]

"This is Unity. It’s like the big brother of Scratch. It’s what many
professional game studios use. You might have played games made with
Unity and not even known it. Among Us — Unity. Cuphead — Unity.
Hollow Knight — Unity."

💡 [DEEPER INSIGHT]:
"Unity uses a programming language called C# (pronounced 'C-sharp').
C# is a high-level language that gives you incredible control. But
with that control comes complexity. In Scratch, you drag a 'move 10
steps' block. In Unity, you might write actual math code to move an
object smoothly. The trade-off is: more power, but more to learn."

"But here’s what’s cool — Unity isn’t just for games. It’s also used for
virtual reality, architecture walkthroughs, film production (like the
Mandalorian virtual sets!), and even self-driving car simulations. So
learning Unity is learning a tool that might one day land you a job
in many different fields."

"Should you start with Unity right now? Probably not while you’re first
learning. But it’s a great goal for the future. Once you’ve mastered the
logic of game engines in Scratch, moving to Unity will feel like leveling
up to a bigger, more powerful world."

SLIDE 9 — GODOT
[Duration: ~4 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Now meet Godot. If Unity is like a giant corporation’s tool, Godot is
like a community-built clubhouse. It’s completely free — no fees, ever.
And its source code is open for anyone to see and modify. That means a
whole world of volunteers improve it every day."

💡 [MAKE IT RELATABLE]:
"Imagine a cookbook. Unity is a fancy cookbook that costs money and you
can’t peek inside the secret recipes. Godot is a cookbook that’s free,
and the recipes are written on the walls for everyone to read and improve.
If you find a way to make a recipe better, you can share it with the world."

"Godot uses a scripting language called GDScript that is designed to be
easy for beginners. It looks a lot like Python, which is one of the most
popular first languages to learn. Many indie developers — small teams or
even one person making a game in their bedroom — choose Godot because it’s
light and runs on almost any computer. Games like Brotato and prototypes
of Celeste were made with Godot."

"For a young learner who is ready to go beyond Scratch but not ready for
the complexity of Unity, Godot is a perfect next step."

SLIDE 11 — COMPARISON: SCRATCH vs UNITY vs GODOT
[Duration: ~3 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"Unity: professional 2D and 3D, huge asset store with pre-made parts,

"Godot: completely free, lightweight, uses beginner-friendly GDScript.

💡 "Choosing the right engine is like choosing the right vehicle. A

SLIDE 12 — HANDS-ON ACTIVITY: LET'S BUILD A SIMPLE SCRATCH GAME
[Duration: ~12 minutes]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

"This is it — the lab portion of our lesson. You’re going to build a
tiny game right now. Follow these steps on your laptop. If you don’t
have a laptop, pair up with someone who does."

[Walk through the timeline steps one at a time. Pause between each to
allow students to follow along.]

"STEP 1: Open scratch.mit.edu and click 'Create.' Choose any sprite —
the cat is fine, or pick a character you like."

[Wait 60 sec, circulate]

"STEP 2: From the Events category, drag out 'when green flag clicked.'
Then from Control, attach a 'forever' block. Inside forever, put an
'if key pressed' block (sensing), and if right arrow, add a 'change x
by 10' from Motion. Do the same for left arrow (change x by -10), up
and down (change y by 10 / -10)."

[Walk around, assist. This is the core coding moment.]

"STEP 3: Add a second sprite — a star or a ball — something to collect.
We’re going to add a score system. Make a variable called 'Score' (Data
category). Then in your player sprite, inside the forever, add: 'if
touching [treasure] then change Score by 1 and go to random position.'
Play! Does the score go up?"

"STEP 4: Test your game. Does the sprite move? Can you make it go
faster? Can you make the treasure harder to catch? Try adjusting the
numbers. Remember iterative design: test, think, improve. You might
even add a sound effect from the Sound category."

✋ [Circulate actively, ask questions: 'What happens if you change the
   speed? What could you add to make it more fun?']

💡 "Notice what you just did. You set up interaction between sprites
(detecting touching), you used variables (score), and you controlled
movement with keyboard input — all with just a handful of blocks. This
is a prototype. In a real studio, they’d do the same thing, then test
it with players, then improve it again and again."

APPENDIX A — COMMON PITFALLS & HOW TO HANDLE THEM
================================================================================

PITFALL: Student says "I can't code" or "This is too hard."
  Handle: "That's exactly how every game developer felt the first time
          they opened a tool. Start by copying the blocks I show you.
          Then change one number. You're already coding."

PITFALL: Student's Scratch sprites don't move correctly.
  Handle: Ask them to check: "Is 'when green flag clicked' at the top?
          Is the movement block inside a forever loop? Did you use the
          right key in the 'if key pressed' block? Debugging is part of
          learning — even pros spend half their time fixing bugs."

PITFALL: Student says "Why use Unity or Godot when Scratch is easier?"
  Handle: "Great question. Scratch is perfect for learning the logic.
          But if you ever want to make a 3D world, or a game you can
          sell, or something that runs on a console, you'll need the
          power of Unity or Godot. It's like learning to ride a bike
          before you drive a car."

PITFALL: Student doesn't see the point of iterative design.
  Handle: "Think of your favorite game. Now imagine the very first version
          the creators made. I guarantee it was ugly, boring, and might
          not have even worked. The game you love is version 100. That's
          iterative design — polishing a rock until it's a diamond."

PITFALL: Student is frustrated that their game doesn't look professional.
  Handle: "Professional-looking games take teams of artists years to make.
          Your job right now is not to make a perfect game — it's to make
          something that works and have fun doing it. Every game artist
          started with a stick figure."

APPENDIX B — EXTENSION QUESTIONS FOR CURIOUS MINDS
================================================================================

1. Could you write your own game without an engine?
   Answer: Yes, but you'd have to write DirectX or OpenGL code for
   graphics, implement physics from scratch, and manage sound drivers. An
   engine saves literally years of work. However, building a tiny game
   from scratch in Python with a simple library is a great way to
   understand what engines do under the hood.

2. What’s the difference between a game engine and a game framework?
   Answer: An engine is a complete tool with an editor, asset manager, and
   built-in systems (like Unity). A framework is a lighter set of code
   libraries you use to build things more manually (like Phaser or
   Pygame). Engines give you more pre-made structure; frameworks give you
   more control.

3. Why are there so many game engines? Why not just one?
   Answer: Different engines serve different needs — 2D vs 3D, small team
   vs large studio, simple vs complex. Also, competition drives
   innovation. Godot’s open-source freedom pushes Unity to stay
   accessible, and vice versa.

4. How do you become a game engine developer?
   Answer: Start by using engines, then learn C++ (many engines are built
   in it), study computer graphics, and contribute to open-source engines
   like Godot. It’s a big goal, but beginning with Scratch is literally
   step one on that path.

5. Can AI make game engines obsolete?
   Answer: No. AI can help generate code or art, but engines solve deep
   computational problems — real-time physics, efficient rendering,
   cross-platform deployment. AI might assist, but the fundamental need
   for an engine remains. In fact, many AI training simulations use game
   engines, so they work together.
